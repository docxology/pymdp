# pymdp docxology sidecar

The `docxology/` directory is the **Standalone Validation Ecosystem** for the [pymdp](https://github.com/infer-actively/pymdp) Active Inference repository. It serves to provide standalone, independent confirmation of active inference technical details and enforcing the "Zero-Mock Policy" via real, execution-tested method validation.

By maintaining this separate tracking and orchestration sidecar, the main `pymdp` tree and its lightweight testing suites can track upstream cleanly while this orchestrator independently drives deeper, resource-intensive validations.

## 1. Subsystem Architecture

The validation architecture is built on a highly configurable, thin-orchestrator pipeline. The folder structure strictly separates configurations, test runners, validation tools, and human-readable documentation.

| Subsystem Component | Role & Functionality                                                                                                                                                                                                                                            |
| ------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **`run_all.py`**    | **The Top-Level Pipeline Orchestrator**: Wraps `pkg.support.mirror_dispatch` to systematically run every execution target registered via manifestations. It auto-resolves the virtual environment and saves completely structured numeric results to `output/`. |
| **`AGENTS.md`**     | **Machine-Readable Contract**: Sets the boundaries for Agent interaction within this directory, explicitly detailing the mathematical schemas and methods implemented across the example scripts.                                                               |
| **`manifests/`**    | **Test Targets (`.txt`)**: Defines exactly _what_ is executed. Contains pointer lists resolving to specific notebooks or python orchestrator scripts (e.g., `orchestrations.txt`, `notebooks_ci.txt`).                                                          |
| **`pkg/`**          | **Helper Methods & Dispatch Components**: Provides `pkg.support`. Functions here natively pull in real `pymdp` methods. Examples include plotting utilities, metric tracking, and the `mirror_dispatch.py` test harness.                                        |
| **`examples/`**     | **Constructed Validation Workloads**: Directly implements the specific 32 core scenarios testing standard environments, Deep Tree Search (SI / MCTS), state inference algorithms, and Dirichlet learning.                                                       |
| **`scripts/`**      | **Execution Wrappers**: Provides CLI command runners such as `run_docxology_notebooks.py` (which routes to nbval using strict modes) and `run_upstream_test_suite.sh`.                                                                                          |
| **`tests/`**        | **CI Pytest Definitions**: Subdirectory containing strictly scoped `pytest` suites confirming that `manifests/` correctly resolve and optional heavy computational notebook tests (`@pytest.mark.slow`).                                                        |
| **`config/`**       | **Orchestration Settings**: Contains `.yaml` definitions manipulating pipeline behavior—for example, switching between `--fast` smoke testing and rigorous full-depth iterations via `orchestration.yaml`.                                                      |
| **`docs/`**         | **Operator Index**: Houses `examples_catalog.md` and `validation_matrix.md`, linking the output generated by `run_all.py` directly to the official `docs-mkdocs/` theory guides and Sphinx API pages.                                                           |

## 2. Standalone Independent Confirmation

The sidecar achieves **independent confirmation** of technical details by relying entirely on the _exact upstream methods_ exposed inside the main `pymdp/` module without hallucination or mock interfaces.

When you invoke:

```bash
python run_all.py
```

The script systematically loads all scripts defined in `manifests/orchestrations.txt`. Each script utilizes methods specifically imported from the upstream package (e.g., `pymdp.agent.Agent`, `pymdp.control.compute_neg_efe_policy`). The orchestrator's post-processing pipeline then automatically produces 4 standard artifacts per example:

- `{stem}_validation.json` — handler diagnostics
- `{stem}_full_data.json` — complete info dict with derived Shannon entropy and invariant audits
- `{stem}_model_trace.npz` — compressed NumPy archive of all extracted tensor parameters
- `{stem}_execution_report.md` — auto-generated Markdown with config, invariants, Performance Insights, and embedded PNGs

Plus up to 13 conditional visualizations via 21 headless-safe plotting functions. Outputs are organized as `output/{category}/{stem}/`.

This guarantees that documentation generated or verified through this folder is biologically and computationally accurate against the current checked-out SHA of the repo.

## 3. Setup & Invocation

### Option A — repo root environment (recommended)

From the pymdp repository root:

```bash
uv venv .venv
source .venv/bin/activate
uv sync --group test
```

Add extras when you run heavier examples:

```bash
uv sync --group test --extra nb --extra modelfit
```

_Note: Graphviz system libraries are required for `pygraphviz`._

### Option B — sidecar-only environment

```bash
cd docxology
uv sync --group test
```

This installs `inferactively-pymdp` as an editable dependency from the parent directory and pulls lightweight pytest/nbval tooling into the docxology virtualenv.

To match upstream notebook CI:

```bash
cd docxology
uv sync --group test --group notebooks
```

### Execution Pathways

- **Run all independent orchestrator checks:**

  ```bash
  uv run python run_all.py
  ```

- **Run targeted Python orchestrations:**

  ```bash
  uv run python scripts/run_docxology_orchestrations.py
  ```

- **Run notebook tests via nbval driver:**

  ```bash
  uv run python docxology/scripts/run_docxology_notebooks.py
  ```

- **Run docxology pipeline structural tests:**
  ```bash
  uv run pytest
  ```

## CI & Deployment Note

GitHub Actions only loads workflows from the repository root `.github/workflows/`. Running docxology validation in hosted CI requires a root workflow or an external pipeline that invokes the commands above. `docs-mkdocs/tutorials/notebooks.manifest` may list notebooks that are not present in every checkout. **This tree** is mathematically authoritative for local continuous tests.
